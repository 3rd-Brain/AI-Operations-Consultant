# Navigator

A local **wiki + dashboard** over an AI-Ops engagement library — the visual companion
to the `ai-ops` skill. Read the library as rendered pages, edit any `.md` in place,
create new stub pages from templates, and see the org chart and per-workflow step maps.
Everything it displays is derived live from the library files — the navigator carries
no assessment state of its own. Zero npm dependencies — Node built-ins only (Mermaid +
marked load from CDN, so first paint needs internet).

Division of labor: the **navigator** visualizes, edits, and creates stubs; **Claude Code**
(the `ai-ops` consultant) does the enriching and interviewing.

The `ai-ops` skill launches this for the operator; it can also be run standalone.

## Run

The navigator serves whatever folder is `ROOT` (defaults to the current working
directory). Point it at the engagement library (the `{{output_root}}` the skill writes to):

```bash
ROOT="<company-library-dir>" node skills/ai-ops/scripts/navigator/server.mjs --port 4173
# then open http://localhost:4173
```

Or from inside the library folder: `node /path/to/skills/ai-ops/scripts/navigator/server.mjs`.

It binds to `127.0.0.1` only and rejects requests with a non-localhost `Host` header or
a cross-origin `Origin` header on writes. It reads/writes strictly inside `ROOT`; only
`.md` files are writable, and the `.ai-ops/` control folder is read-only here (it belongs
to the consultant). Stop with Ctrl-C.

## Views

| View | What it does |
|------|--------------|
| **🏠 Dashboard** | Library counts (roles / workflows / tools), library completeness (stub vs enriched, detected from content), the profile's own Maturity signal prose, and top pains from `.ai-ops/pains-and-bottlenecks.md` — all derived live from the files. |
| **👥 Org & Roles** | Org chart from each role's `**Reports to:**`, with workflows owned per role (from each workflow's `**Owner role:**`) and a by-department roll-up. |
| **🔀 Workflow Maps** | Each workflow as a step flow, color-coded by role and tool, with exception / escalation paths. |
| **📄 Pages** | Every `.md` as a rendered page (mermaid blocks render as live diagrams), with a Read/Edit toggle that saves to disk. |
| **➕ New** | A `+` on each creatable section heading (roles / tools / workflows / decisions / scopes / roadmaps) drops an inline title field and writes a new stub from a built-in template (same shapes as the skill's `templates/`). |

## Conventions it reads

- **Org chart** — role pages carry `**Department:**` and `**Reports to:** <name>`.
- **Workflow ownership** — workflow pages carry `**Owner role:** [name](../roles/x.md)`.
- **Workflow steps** — the `## SOP` section in the skill's template format, one step per line:
  ```
  - [ ] **<Action verb>** — <role> uses <tool> on <record> — <produces what>
    - Exception: <what can go wrong> → <how it's handled>
    - Escalate: <condition> → <who/where>
  ```
  `<role>` and `<tool>` may be plain names or markdown links like
  `[tool-slug](../tools/tool-slug.md)`; the `on <record>` clause is optional.
  `- Exception:` / `- Escalate:` sub-bullets attach to the step above them as
  exception paths. (The legacy `` - [ ] **Action** — `Role` uses `Tool` — note ``
  backtick format still parses for older libraries.)
- **Stubs** — a page is flagged as a stub when it carries a `Stub generated by …`
  marker comment (written by both library assembly and the navigator's ➕ flow) or
  is under ~200 characters.
- **System map** — a ` ```mermaid ` block in `data-architecture.md` renders as a live
  diagram when the page is opened.
